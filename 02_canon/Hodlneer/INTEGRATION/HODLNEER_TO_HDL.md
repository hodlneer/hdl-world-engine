# Hodlneer → HDL Integration

## Purpose
This document defines the canonical integration path from Hodlneer into HDL.fun. It explains: why a Hodlneer user would be encouraged to join HDL.fun, when and where that prompt should appear, what HDL.fun adds beyond the standalone Hodlneer app, what data is shared across products, what must remain private, and how the relationship remains strategically connected without making Hodlneer dependent on HDL.fun.

See also: [[HODLNEER_INDEX]] | [[HDL_TO_HODLNEER]] | [[IDENTITY_MODEL]] | [[PRODUCT_PRINCIPLES]]

---

## Core Integration Principle

> Hodlneer proves your identity.
>
> HDL.fun gives that identity more places to live, move, and matter.

Hodlneer is the origin authority. HDL.fun is the amplification layer.

---

## Why a Hodlneer User Would Join HDL.fun

A Hodlneer user has already received value from: verification, badge reveal, wallpaper, shareability, portable identity.

So the HDL prompt must answer: *"Why should I do anything beyond this?"*

The answer is: HDL.fun gives the badge more context, more utility, and more living presence.

---

## Canonical HDL Value Adds

When a user brings their Hodlneer identity into HDL.fun, HDL may provide:

### 1. Richer Identity Display
The badge can live inside: a profile surface, wallet surfaces, social/post surfaces, community presence layers, future 3D/VR identity environments.

### 2. Social Visibility
The badge can become part of: how others discover or interpret the user, trust and context in interactions, visible reputation signaling.

### 3. Wallet Utility
The user can move from static identity proof into: wallet-linked utility, sending/receiving experiences, social-financial interactions, handle-based identity utility.

### 4. Deeper Ecosystem Presence
The badge becomes part of a larger living environment instead of only a standalone personal artifact.

### 5. Future Unlocks
HDL may later add: enhanced profile treatment, special display modes, community-gated experiences, higher fidelity world representation, richer identity-linked permissions or features.

These are ecosystem extensions, not requirements for Hodlneer to matter.

---

## Important Strategic Rule

HDL.fun should **not** be positioned as:
- the thing that makes the badge "real"
- the thing that finally gives Hodlneer value
- the mandatory home of the badge

The right positioning is: *Hodlneer is already real on its own. HDL is where that identity can do more.*

---

## User Journey: Hodlneer → HDL

### Primary Trigger Moment
The best moment to introduce HDL is **after the reveal** — when the user has just experienced emotional payoff, now cares about the badge, and is most receptive.

### Recommended Post-Reveal Sequence
1. User completes reveal
2. User sees badge in final idle state
3. User is offered primary immediate actions: Set as Wallpaper / Share Your Reveal / Continue
4. After "Continue" or at the right post-reveal moment, user sees an HDL invitation module with clear explanation of what HDL adds and an optional connect/join path

**Do not interrupt the reveal climax with a cross-sell. That would be tacky.**

---

## Canonical Prompt Positioning

The HDL bridge should feel like: an unlock, an expansion, a next layer, a broader place for the identity to live.

It should **not** feel like: a bait and switch, an upsell ambush, a signup trap, or "now go finish this elsewhere."

---

## Suggested Core Messaging

**Primary Line:** Bring your badge into HDL.fun.

**Supporting Line:** Your Hodlneer badge already proves who you are. HDL.fun gives it more room to live, move, and be seen.

**Alternate Lines:**
- Take your identity beyond the wallpaper.
- Unlock the next layer of your badge.
- Let your Hodlneer identity live inside a larger wallet and social world.

---

## What HDL Should Be Allowed to Inherit

When a user connects Hodlneer to HDL.fun, HDL may inherit:

### Public Identity Data
- Era classification
- Verification classification
- Public badge label
- Badge visual state
- Any public-safe presentation metadata

### Optional User-Approved Data (only if explicitly approved)
- Selected display preferences
- Badge presentation style choices
- Optional public profile decorations related to the badge

---

## What HDL Must Not Automatically Inherit

Without clear user consent, HDL must **not** automatically inherit:
- Wallet addresses not intended for public view
- Source exchange data
- Balances
- Raw evidence artifacts
- Exact timestamps
- Private verification material
- Hidden linked account details

**This is non-negotiable. If HDL silently overreaches, trust dies.**

---

## Public/Private Boundary Rule

The badge result may travel. The raw proof does not, unless the user intentionally allows it. This must remain true across all surfaces.

---

## Account Linking Philosophy

The bridge should support the idea that: Hodlneer can have its own onboarding, HDL can have its own onboarding, a user may later connect the two, and eventual SSO/shared identity is possible without erasing product separation.

---

## UX Rules for the Hodlneer-to-HDL Bridge

### Good Bridge UX
- Appears after value is delivered
- Clearly explains HDL benefit
- Respects user momentum
- Is skippable
- Does not reduce confidence in Hodlneer's standalone value

### Bad Bridge UX
- Appears before badge reveal
- Blocks core Hodlneer completion
- Creates cheap fear of missing out
- Implies the badge is incomplete without HDL

---

## Example Hodlneer-to-HDL Flows

### Flow A — Immediate Post-Reveal Prompt
- Reveal completes → user taps Continue
- HDL prompt appears: "Bring your badge into HDL.fun"
- Explains added social/wallet/display value
- Options: Connect to HDL.fun / Maybe Later

### Flow B — Badge Detail Screen Prompt
- User explores badge details
- Sees subtle HDL module: "Use this badge in HDL.fun" / "Unlock richer identity display and wallet utility"

### Flow C — Share/Wallpaper Follow-Up
After user shares or wallpapers the badge:
- "Want your badge to live beyond your phone? Bring it into HDL.fun."

---

## HDL's Responsibility After Connection

Once the badge is connected into HDL.fun, HDL should:
- Display the public identity respectfully
- Preserve badge prestige
- Avoid flattening it into a tiny generic profile badge
- Maintain visual coherence with Hodlneer's artifact system
- Treat the badge as a first-class identity object

HDL should not: trivialize it, override its visual meaning, or repurpose it in ways that contradict Hodlneer canon.

---

## Product Positioning Summary

**Hodlneer gives:** proof, reveal, artifact, wallpaper, standalone identity prestige.

**HDL.fun adds:** social context, wallet utility, richer identity surfaces, community presence, ecosystem expansion.

That division should remain crisp everywhere.

---

## Final Canonical Statement

> Hodlneer must deliver complete standalone value before HDL.fun is introduced.
>
> HDL.fun is the expansion layer, not the justification layer.
