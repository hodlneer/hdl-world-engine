# HDL → Hodlneer Integration

## Purpose
This document defines the canonical integration path from HDL.fun into Hodlneer. It explains: why an HDL.fun user would be encouraged to adopt Hodlneer, what Hodlneer adds to an HDL identity, how Hodlneer upgrades credibility, trust, and expression inside HDL.fun, where the user journey should surface the prompt, what data should and should not flow between products, and how HDL.fun benefits from Hodlneer without reducing Hodlneer to a mere HDL feature.

See also: [[HODLNEER_INDEX]] | [[HODLNEER_TO_HDL]] | [[IDENTITY_MODEL]] | [[PRODUCT_PRINCIPLES]]

---

## Core Integration Principle

> HDL.fun gives a user presence.
>
> Hodlneer gives that presence proof.

HDL can tell the world who a user is in motion. Hodlneer can tell the world how deeply that identity can be trusted and where it came from in crypto history.

---

## Why an HDL.fun User Would Adopt Hodlneer

An HDL user may already have: a handle, a wallet-linked identity, social visibility, community presence, a profile or reputation surface.

So the Hodlneer prompt must answer: *"Why do I need this if I already have HDL?"*

**Answer:** Because HDL gives presence, but Hodlneer gives a verified identity artifact and proof-based credibility that HDL alone should not fake.

---

## Canonical Hodlneer Value Adds for HDL Users

### 1. Verified Crypto Origin
Their place in crypto history becomes more than a vibe or claim.

### 2. Trust Signal
Their public identity gains an objective credibility layer through Era × Verification.

### 3. Premium Badge Artifact
Their HDL identity becomes visually stronger through a high-value badge system that is portable across products and platforms.

### 4. Reveal Moment
They gain the emotional and collectible payoff that HDL alone does not provide.

### 5. Wallpaper and Share Utility
Their identity leaves the platform and becomes something they can show in real life and across social surfaces.

### 6. Ecosystem Prestige
Their HDL identity is now backed by a deeper, standalone identity authority.

These benefits should be clearly articulated. Do not make users guess.

---

## Important Strategic Rule

HDL.fun should not pretend to already be Hodlneer.

That means HDL must **not**:
- Create fake "OG" vibes without verification
- Hand out equivalent prestige treatment without actual proof
- Collapse Hodlneer identity into basic HDL profile cosmetics

If HDL does that, Hodlneer loses meaning and the ecosystem becomes internally dishonest.

---

## Canonical HDL-to-Hodlneer Prompt Positioning

The HDL → Hodlneer bridge should feel like: an upgrade, a proof layer, a credibility unlock, a premium identity enhancement.

It should **not** feel like: a random detour, another signup chore, a forced compliance step, a confusing second app for no reason.

---

## Suggested Core Messaging

**Primary Line:** Verify your origin with Hodlneer.

**Supporting Line:** Your HDL identity shows who you are now. Hodlneer proves where you came from.

**Alternate Lines:**
- Add proof to your HDL identity.
- Turn your handle into a verified identity artifact.
- Unlock the badge behind the identity.

---

## Best HDL Surfaces for the Hodlneer Prompt

The prompt should appear where the user is most likely to understand the value of deeper identity.

### 1. Profile / Identity Surface
If the user is customizing or viewing their HDL identity, this is a natural place to suggest: verify your crypto origin, add your Hodlneer badge, unlock a stronger identity signal.

### 2. Trust-Sensitive Social or Wallet Contexts
If HDL emphasizes trust, visibility, or social interpretation, Hodlneer can be framed as: a stronger trust layer, a deeper reputation marker, a verified identity upgrade.

### 3. Onboarding Completion / Early Journey
Once the user has enough HDL context to understand what identity means, they can be introduced to Hodlneer as: the next layer of proof, a way to strengthen their handle.

*Do not lead with Hodlneer too early if the HDL user has not yet understood HDL itself.*

### 4. Feature Unlock / Badge Placeholder Areas
A tasteful locked or unverified state can invite exploration:
- "Add your Hodlneer badge"
- "Verify your crypto origin"
- "Unlock proof-backed identity"

This must be done with restraint. No cheap manipulation.

## Example HDL-to-Hodlneer Flows

### Flow A — Identity Upgrade Prompt
- User completes HDL profile setup
- HDL shows identity completion screen
- One module offers: "Verify your origin with Hodlneer"
- Explains badge, proof, and trust value
- Options: Get Verified / Maybe Later

### Flow B — Empty Badge Slot
- User sees badge area on profile or wallet surface
- HDL explains: "This slot is reserved for your Hodlneer badge" / "Verify your origin to unlock it"

### Flow C — Trust Enhancement Prompt
- User enters a context where stronger identity matters
- HDL suggests: "Want a stronger proof signal? Verify with Hodlneer."

---

## UX Rules for the HDL-to-Hodlneer Bridge

### Good HDL-to-Hodlneer UX
- Clearly explains the benefit
- Does not interrupt core HDL adoption too early
- Respects the fact that Hodlneer is a separate product
- Creates desire through meaning, not pressure
- Lets the user return later

### Bad HDL-to-Hodlneer UX
- Makes users think HDL is incomplete without Hodlneer
- Appears before HDL identity has any meaning to them
- Overuses locked-state manipulation
- Feels like app sprawl for no reason

---

## What Data HDL May Send Into Hodlneer

To create a smoother bridge, HDL may pass limited user context into Hodlneer if approved and technically appropriate:
- User handle
- Basic account identity needed for linking
- Display preferences relevant to the connection state

This should be done only when useful and privacy-safe.

---

## What HDL Must Not Silently Send

Without clear user understanding and consent, HDL must not silently send:
- Hidden wallet data
- Social activity history intended to remain internal
- Private financial signals
- Private account context unrelated to verification
- Fabricated or inferred timeline claims

The user must understand when they are stepping into verification territory.

---

## Canonical Value Split

**HDL gives:** handle identity, social/wallet presence, ecosystem participation, utility in motion.

**Hodlneer gives:** verified origin, badge artifact, premium reveal, credibility layer, portable proof-based identity.

That split is the whole point.

---

## Post-Verification Return Path

After an HDL user completes Hodlneer verification, the return to HDL should feel rewarding.

**HDL should receive:** public badge identity result, badge display state, optional approved presentation preferences.

**HDL should then:** display the badge proudly, preserve its visual prestige, reinforce that the user's identity has been upgraded with proof, avoid flattening it into a tiny meaningless icon.

---

## Final Canonical Statement

> HDL.fun should make users want Hodlneer because Hodlneer upgrades identity with proof.
>
> HDL provides presence.
>
> Hodlneer provides proven origin.
