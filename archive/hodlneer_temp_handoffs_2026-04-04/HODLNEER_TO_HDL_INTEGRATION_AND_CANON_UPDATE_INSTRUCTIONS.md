## Why a Hodlneer User Would Join HDL.fun

A Hodlneer user has already received value from:
- verification
- badge reveal
- wallpaper
- shareability
- portable identity

So the HDL prompt must answer a simple question:

> “Why should I do anything beyond this?”

The answer is:
HDL.fun gives the badge more context, more utility, and more living presence.

---

## Canonical HDL Value Adds

When a user brings their Hodlneer identity into HDL.fun, HDL may provide:

### 1. Richer Identity Display
The badge can live inside:
- a profile surface
- wallet surfaces
- social/post surfaces
- community presence layers
- future 3D/VR identity environments

### 2. Social Visibility
The badge can become part of:
- how others discover or interpret the user
- trust and context in interactions
- visible reputation signaling

### 3. Wallet Utility
The user can move from static identity proof into:
- wallet-linked utility
- sending/receiving experiences
- social-financial interactions
- handle-based identity utility

### 4. Deeper Ecosystem Presence
The badge becomes part of a larger living environment instead of only a standalone personal artifact.

### 5. Future Unlocks
HDL may later add:
- enhanced profile treatment
- special display modes
- community-gated experiences
- higher fidelity world representation
- richer identity-linked permissions or features

These are ecosystem extensions, not requirements for Hodlneer to matter.

## Important Strategic Rule

HDL.fun should not be positioned as:
- the thing that makes the badge “real”
- the thing that finally gives Hodlneer value
- the mandatory home of the badge

That would weaken Hodlneer and make the whole architecture feel dishonest.

The right positioning is:
- Hodlneer is already real on its own
- HDL is where that identity can do more

## User Journey: Hodlneer → HDL

### Primary Trigger Moment
The best moment to introduce HDL is **after the reveal**.

Why:
- the user has just experienced emotional payoff
- they now care about the badge
- they are most receptive to learning what else it can do

### Recommended Post-Reveal Sequence
1. User completes reveal
2. User sees badge in final idle state
3. User is offered primary immediate actions:
   - Set as Wallpaper
   - Share Your Reveal
   - Continue

4. After “Continue” or at the right moment in the post-reveal flow, the user sees:
   - an HDL invitation screen or module
   - clear explanation of what HDL adds
   - an optional connect/join path

This timing matters.
Do not interrupt the reveal climax with a cross-sell. That would be tacky.

## What HDL Should Be Allowed to Inherit

When a user connects Hodlneer to HDL.fun, HDL may inherit:

### Public Identity Data
- era classification
- verification classification
- public badge label
- badge visual state
- any public-safe presentation metadata

### Optional User-Approved Data
Only if explicitly approved:
- selected display preferences
- badge presentation style choices
- optional public profile decorations related to the badge

---

## What HDL Must Not Automatically Inherit

Without clear user consent, HDL must not automatically inherit:
- wallet addresses not intended for public view
- source exchange data
- balances
- raw evidence artifacts
- exact timestamps
- private verification material
- hidden linked account details

This is non-negotiable.
If HDL silently overreaches, trust dies.

---

## Public/Private Boundary Rule

The badge result may travel.
The raw proof does not, unless the user intentionally allows it.

That must remain true across all surfaces.

---

## Account Linking Philosophy

The bridge should support the idea that:
- Hodlneer can have its own onboarding
- HDL can have its own onboarding
- a user may later connect the two
- eventual SSO/shared identity is possible, but should not erase product separation

### Long-Term Direction
A shared account relationship may eventually allow:
- identity continuity
- smoother onboarding
- cross-product recognition
- easier transitions between products

But the products should still remain conceptually separate.

---

## UX Rules for the Hodlneer-to-HDL Bridge

### Good Bridge UX
- appears after value is delivered
- clearly explains HDL benefit
- respects user momentum
- is skippable
- does not reduce confidence in Hodlneer’s standalone value

### Bad Bridge UX
- appears before badge reveal
- blocks core Hodlneer completion
- creates fear of missing out in a cheap way
- implies the badge is incomplete without HDL

## Example Hodlneer-to-HDL Flow

### Flow A — Immediate Post-Reveal Prompt
- Reveal completes
- User lands on badge result
- User taps Continue
- HDL prompt appears:
  - “Bring your badge into HDL.fun”
  - explains added social/wallet/display value
  - options:
    - Connect to HDL.fun
    - Maybe Later

### Flow B — Badge Detail Screen Prompt
- User explores badge details
- sees subtle HDL module:
  - “Use this badge in HDL.fun”
  - “Unlock richer identity display and wallet utility”

### Flow C — Share/Wallpaper Follow-Up
After the user shares or wallpapers the badge, they may be prompted:
- “Want your badge to live beyond your phone? Bring it into HDL.fun.”

This is a softer secondary path.

## HDL’s Responsibility After Connection

Once the badge is connected into HDL.fun, HDL should:
- display the public identity respectfully
- preserve badge prestige
- avoid flattening it into a tiny generic profile badge
- maintain visual coherence with Hodlneer’s artifact system
- treat the badge as a first-class identity object

HDL should not:
- trivialize it
- override its visual meaning
- repurpose it in ways that contradict Hodlneer canon
