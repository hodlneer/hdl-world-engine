# HODLNEER_ASSET_PRODUCTION_CHECKLIST_AND_CANON_UPDATE_INSTRUCTIONS.md

This document defines the **asset production checklist** for the standalone **Hodlneer** product vertical.

It exists to turn the current canon, launch strategy, screenshot plan, and App Store work into a practical production checklist so that launch execution does not become chaotic, incomplete, or inconsistent.

This document defines:
- which core assets must exist before launch
- which assets are optional but strategically valuable
- how assets should map back to product truth
- what teams need to coordinate
- what approval logic should be used
- what must never be skipped just because people got excited and tried to launch on vibes alone

This is not just a list of files.
It is the production discipline layer.

## Core Production Principle

> Hodlneer should launch like a product that knows exactly what it is.

That means every launch-facing asset should reinforce the same truths:
- proof
- badge desirability
- reveal payoff
- wallpaper value
- privacy confidence
- optional HDL expansion

If the asset stack tells five different stories, launch quality collapses.

---

## Asset Priority Levels

Assets should be grouped by importance.

### Tier 1 — Must Exist Before Public Launch
These assets are mandatory.

### Tier 2 — Strongly Recommended
These assets are not strictly required to ship, but materially improve clarity, conversion, or trust.

### Tier 3 — Nice to Have
These assets can follow after launch if necessary, but should already be envisioned.

# Tier 1 — Must Exist Before Public Launch

## 1. Final Product Positioning One-Liner
**Purpose:**  
Ensure every team can answer what Hodlneer is in one sentence.

**Required Outcome:**  
A single approved line, for example:

**Owner(s):**
- product
- copy

No launch without this being locked.

# Production Workstreams

Launch assets should be coordinated across the following lanes.

## Copy Lane
Owns:
- landing page
- App Store copy
- announcement copy
- FAQ
- HDL bridge messaging

## Visual Design Lane
Owns:
- screenshot system
- app icon
- badge hero renders
- wallpaper presentations
- supporting launch graphics

## Motion / 3D Lane
Owns:
- reveal teaser
- app preview/trailer
- badge motion outputs
- wallpaper motion or showcase assets

## Product / Canon Lane
Owns:
- identity model alignment
- privacy clarity
- message hierarchy enforcement
- approval against canon

## Ecosystem Lane
Owns:
- HDL integration presentation
- connected visuals
- cross-promo alignment

---

# Approval Logic

Every core asset should pass these checks before approval.

## Product Truth Check
Does it reinforce:
- proof
- badge
- wallpaper
- privacy
- optional HDL next layer

## Canon Check
Does it align with:
- identity model
- product principles
- badge visual system
- launch strategy

## Emotional Check
Does it feel:
- premium
- clear
- worth carrying

## Anti-Bullshit Check
Does it avoid:
- crypto cringe
- overhype
- fake luxury
- HDL confusion
- generic app-store blandness

If an asset fails those checks, it is not ready.

# What Good Looks Like

A strong production state should feel like:
- every launch-facing asset tells the same story
- the badge feels like the hero
- wallpaper feels real, not tacked on
- privacy is clear
- HDL is integrated without hijacking the message
- the product feels like it knows itself

---

# What Must Never Happen

The asset pipeline must never devolve into:
- launch-by-vibes
- screenshot chaos
- random copy variations that contradict each other
- HDL-centric asset drift
- generic crypto visuals replacing badge-world coherence
- the reveal being underproduced because “we’ll fix it later”

That “we’ll fix it later” line is how launches get ugly fast.
