# Admin Review Workflow Spec

## 1. Purpose

This document defines the admin review layer for the Hodlneer whitelist recognition pipeline. It translates the policy in `WHITELIST_SYSTEM_SPEC.md` into an operational workflow: who reviews, what they see, what actions they take, and what happens as a result.

This spec is the bridge between the capability spec (what the system does) and the backend/admin UI implementation.

---

## 2. Scope

This workflow applies to all candidates generated by the whitelist matching engine. It does NOT cover:

- Standard badge tier assignment for Bronze/Silver/Gold (driven by onboarding proof engine)
- User-initiated badge upgrades outside the whitelist path
- Manual admin entry of Platinum candidates who were never surfaced by the matching engine

Those paths require separate specs.

---

## 3. Review Queue States

| State | Internal ID | Meaning |
|-------|-------------|---------|
| Pending Review | `pending_review` | Match candidate detected by engine; awaiting admin assignment and evaluation |
| In Review | `in_review` | A reviewer has claimed the case and is actively evaluating it |
| Needs More Info | `needs_more_info` | Reviewer has requested additional evidence before deciding |
| Approved | `approved` | Badge assignment authorized; badge generation triggered |
| Rejected | `rejected` | Candidate does not meet criteria; no badge; case closed |
| Escalated | `escalated` | Case flagged for senior review or Core Canon Leadership decision |

State transitions:

```
pending_review
    → in_review         (reviewer claims case)
    → needs_more_info   (from in_review)
    → approved          (from in_review or needs_more_info)
    → rejected          (from in_review or needs_more_info)
    → escalated         (from in_review)
escalated
    → approved
    → rejected
    → needs_more_info
needs_more_info
    → in_review         (evidence submitted; returned to queue)
```

---

## 4. Reviewer Roles

| Role | Access | Can Approve | Can Reject | Can Escalate |
|------|--------|-------------|------------|--------------|
| Admin Reviewer | View queue, claim cases, request info | No (without escalation) | Yes | Yes |
| Senior Reviewer | All Admin Reviewer permissions | Yes | Yes | Yes |
| Core Canon Leadership | All permissions, audit log access | Yes | Yes | — |

**Default launch posture:** All approvals require Senior Reviewer or above. Admin Reviewers may reject or escalate; they may not approve unilaterally.

---

## 5. Queue Entry Trigger

A review case is created when:
1. The matching engine detects a candidate during onboarding with a match score ≥ 0.80
2. A user submits a manual self-flag for Platinum consideration (after onboarding)

Each case is unique per candidate identity. If the same underlying entity triggers multiple user accounts, cases are linked but evaluated independently.

---

## 6. Required Evidence at Review Time

Reviewers must evaluate the following before deciding:

### Always Required
- Match source: which whitelist the candidate appeared on (internal only — never exposed to users)
- Match score and match type (exact / alias / handle / fuzzy)
- Years appeared on the source list
- Confidence tier of the whitelist entry (HIGH / MEDIUM)
- The user's submitted onboarding identity signals (wallet addresses, connected accounts)
- The user's declared era and verification level from onboarding

### Required for Approval
At least one of the following must be confirmed before approval:

- [ ] Identity corroboration: at least one external signal (public wallet, social handle, published work, or equivalent) that links the applicant to the whitelisted entity
- [ ] Contribution evidence: documented contribution to the crypto ecosystem consistent with the whitelisted person's known work
- [ ] No disqualifying red flags: no evidence of fabricated identity, impersonation, or false signals

### For Needs More Info
The reviewer must specify:
- What type of evidence is missing
- What the candidate must provide (e.g., "proof of wallet address ownership," "link to published work," "third-party attestation")
- A deadline for response (default: 30 days)

---

## 7. Reviewer Actions

### Claim Case
- Reviewer selects an unassigned case from the queue
- Case transitions: `pending_review` → `in_review`
- Case is locked to that reviewer (others can view but not act)

### Approve
- Reviewer confirms required evidence is satisfied
- Triggers badge assignment flow (see Section 9)
- Case transitions: `in_review` → `approved`
- Audit entry created

### Reject
- Reviewer confirms candidate does not meet criteria
- Reason must be recorded (from controlled vocabulary — see Section 8)
- No badge is assigned; case is closed
- Case transitions: `in_review` → `rejected`
- Audit entry created

### Request More Info (Needs More Info)
- Reviewer specifies what additional evidence is required
- A notification is sent to the user (generic — does not reveal the whitelist or the match score)
- User has a defined window to submit additional evidence
- Case transitions: `in_review` → `needs_more_info`
- When evidence is submitted: case returns to `in_review` with the new reviewer notification

### Escalate
- Reviewer determines the case exceeds their authorization level or requires judgment above their role
- Case transitions: `in_review` → `escalated`
- A note explaining the escalation reason is required
- Core Canon Leadership is notified

---

## 8. Rejection Reason Codes

Rejections must use one of the following controlled vocabulary entries:

| Code | Meaning |
|------|---------|
| `identity_not_confirmed` | Cannot link the applicant to the whitelisted entity |
| `insufficient_contribution` | Public record of ecosystem contribution does not meet criteria |
| `impersonation_suspected` | Evidence suggests the applicant is claiming another person's identity |
| `fame_without_contribution` | Known public figure but no genuine ecosystem contribution (anti-gaming rule) |
| `wealth_without_history` | High-value wallet but no verifiable historical participation record |
| `no_response` | Candidate did not respond to needs_more_info request within deadline |
| `other` | Requires a free-text explanation |

---

## 9. Post-Approval Flow

When a case is approved:

1. A badge assignment event is queued for the associated user account
2. The badge assignment engine creates a Platinum badge record tied to the user's identity
3. The badge is applied to the user's profile (display rules per `TIER_SYSTEM.md` and `BADGE_VISUAL_SYSTEM.md`)
4. The user receives a notification: they have been recognized (generic — does not name the whitelist or the specific reason)
5. The review case is marked `approved` and archived in the audit log

**The notification must not:**
- Name the source whitelist
- Reveal the match score
- Describe the internal review process
- Make promises about future recognition

---

## 10. Post-Rejection Flow

When a case is rejected:

1. No badge is assigned or modified
2. If the candidate submitted a self-flag (manual path): they receive a notification that their consideration was completed and no recognition was extended at this time
3. If the candidate was system-detected (matching engine path) and has NOT submitted a self-flag: no user-facing notification is sent
4. The review case is marked `rejected` and archived
5. The candidate may be reconsidered in the future if the admin manually re-opens the case (e.g., if new evidence surfaces or criteria evolve)

---

## 11. Needs More Info Flow (End-to-End)

1. Reviewer sets state to `needs_more_info` with evidence requirement and deadline
2. User receives a notification: "Your application is under review. We've requested additional information to complete our evaluation." *(No whitelist reference, no score exposure)*
3. User submits additional evidence through the designated interface
4. Evidence is attached to the review case
5. Case returns to `in_review` state; reviewer is notified
6. If the deadline passes with no submission: reviewer may reject with code `no_response`

---

## 12. Audit Logging Requirements

Every state transition must produce an immutable audit log entry containing:

| Field | Description |
|-------|-------------|
| `case_id` | Unique review case identifier |
| `timestamp` | ISO 8601 timestamp of the action |
| `action` | Action taken (claim, approve, reject, needs_more_info, escalate) |
| `actor_id` | ID of the admin who performed the action |
| `actor_role` | Role at time of action |
| `from_state` | Previous state |
| `to_state` | New state |
| `reason_code` | For rejections: the controlled vocabulary code |
| `notes` | Free-text notes (required for escalation and needs_more_info; optional otherwise) |
| `evidence_refs` | IDs of any evidence artifacts attached at this step |

Audit logs are:
- Write-once (no update or delete)
- Admin-read-only (not exposed to end users)
- Retained indefinitely (whitelist recognition decisions are permanent record)
- Not subject to user deletion requests (they are operational records, not personal data)

## 13. Data Visibility — Admin vs End User

| Data Point | Admin View | End User View |
|-----------|-----------|--------------|
| Match score | ✅ Full | ❌ Never |
| Source whitelist name | ✅ Full | ❌ Never |
| Years appeared on whitelist | ✅ Full | ❌ Never |
| Confidence tier of entry | ✅ Full | ❌ Never |
| Review case state | ✅ Full | Generic status only ("under review", "complete") |
| Rejection reason code | ✅ Full | ❌ Never |
| Audit log | ✅ Full | ❌ Never |
| Badge assigned (Platinum) | ✅ Full | ✅ Badge visible per user's display settings |
| Identity signals submitted by user | ✅ Full | Own signals only |

**Design principle:** The user experience of the recognition process must feel like a black box. Users know they may be considered for recognition. They do not know how the system works, what lists exist, or how scoring works.

---

## 14. Edge Cases

### Duplicate Accounts
If two user accounts are linked to the same whitelist entity:
- Both are surfaced as candidates
- Cases are linked in the queue
- Only one account may be approved for Platinum (the primary verified identity)
- The other case must be rejected with `identity_not_confirmed` or escalated

### Shared Identity (Collective / Pseudonymous Figures)
- Entities marked `anonymous`, `symbolic`, or `collective` in the whitelist are excluded from matching
- If a user claims to be a known pseudonymous figure (e.g., a Satoshi claimant), the case must be escalated automatically

### User Withdraws Self-Flag
- If a user withdraws their manual Platinum consideration request, the review case is closed without a rejection decision
- The state transitions to `withdrawn` (no audit trail of rejection; no future barrier to re-submission)

## 15. API Endpoints

This spec is implemented via the endpoints defined in `08_bridges/hodlneer/whitelist-api.yaml`:

| Operation | Endpoint |
|-----------|----------|
| View review queue | `GET /whitelist/review-queue` |
| Approve candidate | `POST /whitelist/matches/{id}/approve` |
| Reject candidate | `POST /whitelist/matches/{id}/reject` |
| Flag for more info | `POST /whitelist/matches/{id}/flag` |
| View candidate detail | `GET /whitelist/matches/{id}` |

The API spec defines request/response schemas. This document defines the operational policy those endpoints enforce.
